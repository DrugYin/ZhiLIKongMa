# 代码审查报告（全仓库静态巡检）

> 审查日期：2026-04-23  
> 审查范围：`miniprogram/`、`cloudfunctions/`、`admin-web/src/`（全量 JS/Vue/WXML/WXSS 文件路径遍历）  
> 审查方法：结构化静态阅读 + 关键模式扫描 + 接口清单一致性脚本 + JS 语法批量校验

---

## 一、结论摘要

本次巡检未发现会导致“立即崩溃”的大面积语法问题（100 个 JS 文件语法检查全部通过），但存在多项**高概率线上故障点**与**可维护性隐患**，建议按优先级推进：

- P0（立即处理）
  1. API 层仍引用不存在的云函数（调用必失败）
  2. 班级邀请码存在并发竞争窗口（可能出现重复邀请码）
- P1（本周处理）
  3. 表单组件日期默认值格式不稳定（不同地区格式差异）
  4. 表单组件 `setFormData/resetForm/getErrors` 与当前数据结构不一致（无效代码）
- P2（计划优化）
  5. 云环境 ID 硬编码，缺少多环境配置能力
  6. 上线公告文案长期暴露“数据丢失”提示，影响用户信任

## 二、问题明细与建议

### P0-1：前端 API 声明了不存在的云函数（调用后必失败）
- **位置**：`miniprogram/services/api.js`
- **现状**：仍存在 `get-prizes`、`start-draw`、`get-draw-records`、`get-config` 调用封装；仓库中不存在对应 `cloudfunctions/<name>/index.js`。
- **风险**：一旦页面或业务流程触发，将直接进入失败分支，用户看到“请求失败/网络异常”。
- **建议**：
  - 若功能下线：删除 API 暴露与相关入口；
  - 若功能保留：补齐云函数并同步文档。

### P0-2：`create-class` 邀请码生成存在并发竞争窗口
- **位置**：`cloudfunctions/create-class/index.js`
- **现状**：`createUniqueClassCode()` 通过“先查重再写入”来保证唯一性。
- **风险**：并发创建班级时，两个请求可能同时查到“未占用”并写入相同 `class_code`。
- **建议**（必须至少一条）：
  1. 在数据库层对 `classes.class_code` 建唯一索引；
  2. 对写入冲突错误做捕获重试（指数退避 + 重生成）。

### P1-1：生日选择器默认值使用 `toLocaleDateString()`，格式不可控
- **位置**：`miniprogram/components/form/form.js`
- **现状**：生日选择器 `default-value` 使用 `new Date().toLocaleDateString()` 生成。
- **风险**：不同设备区域设置可能返回 `M/D/YYYY`、`YYYY/M/D` 等格式，与组件声明的 `YYYY-MM-DD` 不一致。
- **建议**：统一手工格式化为 `YYYY-MM-DD`（例如 `padStart` 方案）。

### P1-2：表单组件存在一组与实际状态结构不一致的“死代码”
- **位置**：`miniprogram/components/form/form.js`
- **现状**：
  - `setFormData()` 写入 `formData`，但组件主数据是 `userInfo`；
  - `resetForm()` 重置 `formData/errors/fieldErrors`，这些字段并非稳定参与当前表单流程；
  - `getErrors()` 返回 `fieldErrors`，但未形成与 `rules` 对齐的完整校验闭环。
- **风险**：后续维护者误用这些方法时会得到“看似成功但实际无效”的行为。
- **建议**：
  - 要么删除这组未被实际使用的方法；
  - 要么按 `userInfo` + 明确校验状态重构为可测试逻辑。

### P2-1：云环境 ID 仍硬编码在 `app.js`
- **位置**：`miniprogram/app.js`
- **现状**：`wx.cloud.init({ env: 'zhi-li-kong-ma-7gy2aqcr1add21a7' })`。
- **风险**：多环境切换（dev/staging/prod）容易误连正式环境。
- **建议**：将环境标识迁移至统一配置（构建注入或独立 config）。

### P2-2：启动公告长期包含“存在 bug 和数据丢失”文案
- **位置**：`miniprogram/app.js`
- **现状**：每次启动都可触发升级公告，文案偏“内部排障态”。
- **风险**：对新用户造成稳定性负面感知；且无法灰度/关闭。
- **建议**：
  - 建立可后台配置的公告系统（标题、内容、有效期、是否强制弹出）；
  - 对“风险性文案”设置到期策略。
